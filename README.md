# Claude Code SDLC Scaffold

A generic, autonomous software development scaffold for Claude Code. Simulates a full engineering team for a one-person startup. Write a BRD, approve the specs and architecture, then let the autonomous loop build, test, review, and deploy your app — self-healing from errors and learning from failures along the way.

Inspired by [Karpathy's autoresearch](https://github.com/karpathy/autoresearch) ratcheting pattern and self-improving agent loops.

**This scaffold is domain-agnostic.** All project-specific details come from your BRD.

## Prerequisites

- [Claude Code](https://docs.anthropic.com/en/docs/claude-code) CLI installed
- Python 3.12+ with [uv](https://github.com/astral-sh/uv)
- Node.js 20+ with npm
- Docker and Docker Compose (for local deployment and E2E tests)

## Installation

### Option A: Copy into Your Project

```bash
git clone https://github.com/cwijayasundara/claude_code_forge_v2.git
cp -r claude_code_forge_v2/.claude/ your-project/.claude/
cp claude_code_forge_v2/CLAUDE.md your-project/CLAUDE.md
```

Claude Code automatically loads `.claude/` and `CLAUDE.md` on startup. All commands, agents, hooks, and skills are immediately available.

```bash

```

The scaffold is already active — start writing your BRD and run `/build`.

## How It Works

The scaffold has three operating modes:

**Human-in-the-loop (Phases 1-3):** You write a BRD. `/spec` decomposes it into epics, user stories, and a dependency graph. `/design` generates system architecture and UI mockups. You review and approve both before any code is written.

**Autonomous execution (Phase 4+):** After approval, `/auto` takes over. It picks stories from the dependency graph, spawns implementer agents, runs verification gates, commits on pass, self-heals on failure, and learns from mistakes. You can steer it mid-run by editing `.claude/program.md`.

**Maintenance mode:** Use `/fix-issue`, `/refactor`, and `/improve` for ongoing work against the codebase.

## Pipeline

```
/spec → /design → /implement → /review → /test → /deploy
  │        │          │           │         │        │
  │        │          │           │         │        └─ Docker Compose stack
  │        │          │           │         └─ Playwright E2E + unit/integration
  │        │          │           └─ Code review + security review gate
  │        │          └─ Agent teams for parallel story execution
  │        └─ Architect + UI designer (concurrent)
  └─ BRD → epics, stories, dependency graph
```

**`/build [BRD]`** runs all phases end-to-end. Phases 1-3 pause for human approval. Phase 4+ runs autonomously via `/auto`.

## Architecture

The scaffold enforces a strict layered architecture with downward-only dependencies:

```
UI (React)  →  API (FastAPI)  →  Service  →  Repository  →  Config  →  Types
```

Upward imports are blocked by hooks at edit time and at commit time. You cannot bypass them.

Project-specific architecture (API contracts, data models, folder structure) is generated by `/design` from your BRD and written to `specs/design/`.

## Autonomous Execution

After you approve specs and architecture, `/auto` runs autonomously:

```
Read program.md → Pick story → Implement → Ratchet gate → Pass? → Commit → Next
                                                         → Fail? → Self-heal → Retry
                                                                  → 3rd fail? → Revert + Learn + Escalate
```

### Ratcheting Rules

Only improvements persist. The system never goes backward:

- Coverage never drops below the previous commit's level
- Tests never break — failed iterations revert completely
- Lessons accumulate in `learned-rules.md` — rules are never deleted
- Self-heal first, revert last — always try targeted fixes before throwing away work

### Self-Healing

When verification fails, the agent doesn't immediately revert. It diagnoses the error, classifies it (lint, type error, test failure, import error, architecture violation, coverage drop, runtime error), and applies a targeted fix. Only after 3 failed fix attempts does it revert and escalate.

### Self-Improvement

The scaffold gets better over time through three mechanisms:

1. **Per-project:** `learned-rules.md` accumulates defensive rules extracted from failures. Each rule is injected into every future implementer prompt.
2. **Per-project:** The code-reviewer validates against learned rules and adds new ones when it finds violations. Failures become prevention.
3. **Cross-project:** Promote recurring learned rules to SKILL.md Gotchas sections, making them permanent for all future projects.

### Steering the Loop

Edit `.claude/program.md` while `/auto` is running. The agent re-reads it every iteration.

- **Instructions** — what to build (change focus mid-run)
- **Constraints** — what not to touch (add guardrails)
- **Stopping criteria** — when to halt (adjust thresholds)
- **Pipeline status** — track per-phase progress

## Enforcement Hooks

Quality and security are enforced automatically — not by documentation, but by hooks that block violations:

| Hook | Trigger | Action |
|------|---------|--------|
| `scope-directory.js` | Edit/Write | **BLOCKS** writes outside project directory |
| `protect-env.js` | Edit/Write | **BLOCKS** .env modifications (secrets are human-only) |
| `detect-secrets.js` | Edit/Write | **BLOCKS** hardcoded API keys, passwords, PII |
| `lint-python.js` | Edit/Write .py | Auto-fixes with ruff |
| `typecheck-ts.js` | Edit/Write .ts/.tsx | Runs tsc --noEmit |
| `check-architecture.js` | Edit/Write .py | **BLOCKS** upward layer imports |
| `check-function-length.js` | Edit/Write .py/.ts | Warns on functions >50 lines |
| `check-file-length.js` | Edit/Write .py/.ts | Warns at 200, **BLOCKS** at 300 lines |
| `protect-pdfs.js` | Write to docs/ | **BLOCKS** modification of fixtures |
| `pre-commit-gate.js` | git commit | **BLOCKS** commits with architecture violations |
| `task-completed.js` | Task done | Reminds to run /review |

All hooks are cross-platform Node.js (works on macOS, Linux, and Windows).

## Six Quality Principles

Every agent follows these (from Steve Krenzel's "AI is forcing us to write good code"):

1. **Small, well-scoped modules** — one file = one responsibility
2. **Static typing everywhere** — type-annotate all functions, zero `any` in TypeScript
3. **Functions under 50 lines** — decompose into named subfunctions
4. **Explicit error handling** — typed error classes, no bare exceptions
5. **No dead code** — every line traces to a story
6. **Self-documenting** — good names over comments, types as documentation

## Adapting for Your Project

This scaffold is generic by design. Project-specific details come from your BRD:

1. **Write a BRD** — describe your project, users, workflows, and constraints.
2. **Run `/spec`** — the spec-writer agent decomposes it into epics and stories.
3. **Run `/design`** — the architect generates API contracts, data models, and folder structure. The UI designer creates mockups.
4. **Approve and go** — `/auto` handles the rest autonomously.

To customize the scaffold itself:

- **Adjust architecture** — edit `.claude/architecture.md` to change the layer structure (e.g., add a Cache layer or rename Service to Domain).
- **Add domain skills** — create a new reference skill in `.claude/skills/` with domain-specific patterns (e.g., `.claude/skills/ml-pipeline/SKILL.md` for ML projects).
- **Modify templates** — update deployment templates, page templates, and story templates for your tech stack.
- **Keep the patterns** — the autonomous loop, self-healing, ratcheting, and learned rules are domain-independent. They work for any project.
